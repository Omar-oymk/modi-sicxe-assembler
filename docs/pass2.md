# Pass 2 - Architecture & Flow

Pass 2 is responsible for translating the intermediate representation generated by Pass 1 into binary SIC/XE object code. It performs all bitwise logic for addressing modes and formats the final HTME object file.

## 1. Core Modules Involved
- **`core/pass_2/pass2.py`**: The main orchestrator that parses the output files of Pass 1 and loops through the intermediate instructions.
- **`core/pass_2/assemble_line.py`**: The object code generator that acts as a router, handling formats 1, 2, 3, and 4.
- **`core/pass_2/flags_nixpbe.py`**: Performs advanced address calculations (PC/Base relative) and constructs the final bit sequences.
- **`core/pass_2/htme/`**: Contains scripts to format Header, Text, Modification, and End records.

## 2. Detailed Execution Flow

### Step 1: Loading Pass 1 Data
Pass 2 begins by reading the files generated by Pass 1:
- The Intermediate file (`intermediate.txt`)
- The Symbol Table (`symbTable.txt`)
- The Block Table (`blockTable.txt`)
- The Pool Table (`PoolTable.txt`)

### Step 2: Line-by-Line Assembly & Error Checking (`assemble_line.py`)
For each valid instruction in the intermediate file, the assembler must generate its corresponding hexadecimal object code. During this process, `resolve_operand` actively validates that all symbols used as operands exist in the Symbol Table. If an `Unidentified Symbol` is detected, it raises a formatted `ValueError` which instantly halts Pass 2 and dumps the error details (with the current PC) to `error.txt`.

- **Format 1:** Directly outputs the 1-byte Opcode (`opcode:02X`).
- **Format 2:** Maps register operands (A, X, L, B, S, T, F, PC, SW) to their respective 4-bit integer IDs, packing them into a 2-byte object code.
- **Format 3 (Relative Addressing):**
  - **`n` and `i` flags:** Evaluates the operand prefix (`@` sets `n=1, i=0`, `#` sets `n=0, i=1`, default is `n=1, i=1`).
  - **`x` flag:** Set to `1` if the operand ends in `,X`.
  - **Addressing Calculation:** Pass 2 first calculates the target's absolute memory address (Target Address). It then calculates the Program Counter (`PC = Instruction LC + 3`).
  - **PC-Relative:** Evaluates if `Target - PC` is within the bounds of `-2048` to `2047`. If so, `p=1, b=0`. Negative displacements are wrapped into 12-bit two's complement.
  - **Base-Relative:** If PC-relative fails, it calculates `Target - BASE`. If within `0` to `4095`, `p=0, b=1`. (Requires `BASE` directive to have been set).
- **Format 4 (Extended Addressing):**
  - Used when the instruction is prefixed with `+`.
  - The `e` flag is set to `1`.
  - The literal 20-bit absolute target address is embedded directly into the 4-byte object code.

### Step 3: Literal Pool Variables
If the operand is a literal pool variable (prefixed with `&`), Pass 2 queries the Pool Table for the literal's absolute memory address and resolves it using standard PC-relative or Base-relative addressing like any normal symbol. If the literal's assigned address is too far to be reached by either addressing mode, the assembler immediately raises a formatted `POOLVAR Error` via `ValueError` and writes it to `error.txt`.

### Step 4: Writing HTME Records
Once all object codes are generated, the HTME output is assembled:
1. **Header Record (`H`)**: Extracted from the `START` directive, containing the Program Name, Starting Address, and total size (sum of all blocks).
2. **Text Records (`T`)**: The generated object codes are packed into T records. A new T record is started if:
   - The contiguous size exceeds 30 bytes (60 hex chars).
   - A `RESW` or `RESB` directive interrupts the code.
   - The code switches to a different Block (`USE`).
   - *Note: Pool variables are serialized and appended as Text records representing the data initialized in the POOL block.*
3. **Modification Records (`M`)**: Generated for *every* Format 4 instruction that references memory, telling the system loader to relocate the absolute 20-bit address field upon loading.
4. **End Record (`E`)**: Extracted to mark the starting execution point.

### Step 5: Final Output
The final files `out_pass2.txt` and `HTME.txt` are written to the `output/` directory, completing the assembly process.
